数据模型

这个系统的数据是怎么组织的

数据库，就像图书馆的档案室

想象食堂管理处有一个大型档案室，里面摆着几排档案柜。每个抽屉专门存放一类信息，抽屉之间通过编号相互引用。

🗄️
档案室 = 整个数据库
🗂️
每个抽屉 = 一张数据表
📄
每份文件 = 一条记录（一行）
🔖
文件上的标签 = 字段（列）

这个系统一共有 5 个抽屉，每个抽屉负责一类数据，抽屉之间通过 外键 彼此关联。

五张表，各司其职

五张表之间的关系如下：持卡人拥有一张就餐卡，卡上记录所有的存款和消费流水。

👤

CardHolder

持卡人：id · 姓名 · 证件号 · 创建时间

💳

Card

就餐卡：16位卡号 · 押金 · 余额 · 状态

💰

DepositRecord

存款记录：金额 · 时间

🍜

Transaction

消费记录：金额 · 窗口 · 时间

🏪

Window

售饭窗口：id · 名称

💡
关键设计约束

一人一卡（证件号唯一）· 金额用整数存分（避免浮点精度问题）· 卡号16位随机数（不暴露自增ID）

读懂一张表的代码

下面是 Card（就餐卡）表在 Go 代码中的定义。左边是代码，右边是逐行的中文解释。

CODE 
type Card struct {
    ID           uint       `gorm:"primaryKey" json:"-"`
    CardNo       string     `gorm:"uniqueIndex;not null;size:16" json:"cardNo"`
    CardHolderID uint       `gorm:"not null" json:"cardHolderId"`
    Deposit      int64      `gorm:"not null;default:0" json:"deposit"`
    Balance      int64      `gorm:"not null;default:0" json:"balance"`
    Status       CardStatus `gorm:"not null;default:'active'" json:"status"`
    CreatedAt    time.Time  `json:"createdAt"`
    UpdatedAt    time.Time  `json:"updatedAt"`
}
中文解释

type Card struct — 定义"就餐卡"这张表的数据结构，每个字段对应表里的一列

ID uint · primaryKey — 数据库内部的主键，自动递增，对外接口隐藏（json:"-"）

CardNo string · size:16 — 对外暴露的 16 位随机卡号，设置唯一索引，保证不重复

CardHolderID uint — 指向持卡人表的外键，把这张卡和它的主人关联起来

Deposit int64 — 押金，单位是"分"，固定为 2000（即 20 元）

Balance int64 — 卡内余额，单位是"分"，每次存款加、消费减

Status CardStatus — 卡的状态：active（正常）/ lost（挂失）/ cancelled（注销），默认 active

CreatedAt / UpdatedAt — 创建时间和最后更新时间，由GORM 自动维护

检验一下

🧪 检验理解

1. 系统为什么用整数（int64）来存储金额，而不是小数（float）？

✅ 正确！浮点数的二进制表示会引入微小误差，在金融场景中累积后会造成账目不准。用"分"作为最小单位，全程整数运算，完全避免这个问题。
❌ 再想想。浮点数的问题是精度，不是大小。试试 B 选项——想想为什么 0.1 + 0.2 在计算机里不等于 0.3。

2. 就餐卡的 CardNo（卡号）为什么用 16 位随机数字，而不直接用数据库自增 ID？

✅ 正确！如果卡号是 0001、0002……攻击者可以轻易猜出其他人的卡号，也能推算出系统规模。随机卡号从根本上消除了这种风险。
❌ 不对。安全才是核心原因。如果卡号 = 自增 ID，别人只要知道自己的卡号，就能猜出所有其他人的卡号。

3. 一个持卡人（CardHolder）最多可以拥有几张状态为 active 的就餐卡？

✅ 正确！系统在数据层面通过唯一约束保证一人一卡，这也是需求中"持有效证件办理就餐卡"的核心规则。
❌ 不对。看看需求描述："学生员工持有效证件到管理处……办理就餐卡"——一人只能对应一张卡，系统通过证件号唯一约束来保证这一点。
