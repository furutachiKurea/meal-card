系统全貌

这个系统是什么、解决什么问题、有哪些角色和入口

你有没有遇到过这些烦恼？

早上八点，你拿着饭卡冲进食堂，窗口阿姨刷了一下——

"余额不足，请充值。"

你懵了。昨天明明还有钱的？卡丢了怎么办？充进去的钱还能退吗？

食堂饭卡管理系统就是为了解决这一系列问题而生的。它把食堂的 六项核心业务 全部数字化，让管理员、窗口、学生都能顺畅协作。

学生刷卡 → 余额扣减 → 流水记录 一次 HTTP 请求 → 数据库事务 → 持久化日志

谁在使用这个系统？

🏢

管理员

负责发卡、为学生存款、处理挂失与注销、查看统计报表。是系统的核心操作者。

🍜

窗口阿姨

通过窗口机验卡、录入餐费、完成结算。每次就餐的最终执行者。

🎓

学生

持卡就餐，刷卡时可看到当前余额。无需登录系统，卡即身份。

一次刷卡消费，幕后发生了什么？

三个模块之间的一次真实对话：

🍜 一次刷卡消费的幕后
前端（窗口机页面） 阿姨输入了 8.5 元，我来发一个请求：POST /api/meal/consume，卡号 + 金额。
后端（Go + Echo） 收到！先查这张卡：状态正常？是本单位的卡？有没有挂失？
数据库（SQLite） 查到了，卡状态 active，余额 42.00 元，未挂失。
后端（Go + Echo） 余额充足，开启 事务 ：余额 42.00 − 8.50 = 33.50，写入消费流水。提交事务。
前端（窗口机页面） 收到响应，显示"消费成功，余额 33.50 元"。就餐完成！

技术栈一览

前后端完全分离，通过 REST API 通信。

后端

🐹Go — 编译型语言，性能好、部署简单
⚡Echo — 轻量级 HTTP 框架，路由清晰
🗄️GORM — ORM 库，操作数据库不写 SQL
📦SQLite — 零配置文件数据库，适合课设

前端

⚛️React — 组件化 UI 库
🐜Ant Design — 开箱即用的 UI 组件库
🚀Vite — 极速构建工具，热更新快
🔀React Router — 前端路由管理
