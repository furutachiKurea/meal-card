分层架构的好处

每一层只和相邻的层打交道：Handler 不直接碰数据库，Service 不知道 HTTP 长什么样。这样改动某一层时，其他层不受影响。就像水务系统中，换了水龙头不需要改水厂。

Screen 2 · 统计的秘密 — 聚合查询

统计服务不创建新数据，它只是从已有的消费记录和充值记录中聚合出数字。下面是「日餐报表」功能的核心代码。

Go func (s *StatisticsService) GetDailyReport(date string) (*DailyReportResult, error) { day, err := time.Parse("2006-01-02", date) if err != nil { return nil, newBizError("日期格式无效") } start := time.Date(day.Year(), day.Month(), day.Day(), 0,0,0,0, time.Local) end := start.Add(24 * time.Hour) totalRevenue, count, windows, err := s.cardRepo.GetDailyReport(start, end) }
逐行解读
第 1 行：定义函数 —— 接收一个日期字符串，返回报表结果
第 2-4 行：把日期文字转换成时间对象；如果格式不对就报错
第 5-6 行：计算这一天的起点（0:00）和终点（次日 0:00）
第 7 行：把时间范围交给 Repository，让它去数据库查出总收入、笔数和各窗口数据

什么是聚合查询？

统计不是把每条数据一条条读出来手动加总，而是让数据库用 SUM、 COUNT、 GROUP BY 一次性算好。就像问全班同学的平均分 —— 你不需要挨个问成绩再手动计算，直接让数据库帮你算。

Screen 3 · 代码翻译 — 前端怎么发 API 请求

前端通过 api.js 中的函数向后端发起请求。每个统计功能对应一个函数。

JavaScript export function getMealRevenue({ startTime, endTime }) { return request( `/api/statistics/meal-revenue ?startTime=${encodeURIComponent(startTime)} &endTime=${encodeURIComponent(endTime)}` ) } export function getActiveBalance() { return request('/api/statistics/active-balance') }
逐行解读
getMealRevenue：获取指定时间段的售饭总收入。通过查询参数告诉后端时间范围
?startTime=...&endTime=...：就像信封上的附加说明 —— 「我只要这个时间范围内的数据」
encodeURIComponent：把时间中的特殊字符（如冒号 :）转成安全格式，避免 URL 被破坏
getActiveBalance：获取流动资金总额。不需要参数 —— 直接查所有活跃卡的余额之和

为什么统计接口都用 GET？

HTTP 中 GET 代表「获取数据」，POST 代表「创建/修改数据」。统计功能只是读取已有数据做汇总，不修改任何东西，所以全部使用 GET 请求。就像去图书馆查资料（GET）和往图书馆捐书（POST）是不同的动作。

系统统计模块的全部 API 一览：

GET /meal-revenue 指定时段售饭总收入
GET /window-revenue 各窗口收入分布
GET /holder-deposits 持卡人存款明细（分页）
GET /active-balance 流动资金总额
GET /daily-report 日餐报表
GET /yearly-report 年餐报表（按月汇总）
