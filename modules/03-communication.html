数据的对话 — 前后端如何通信

前端和后端之间的每一次交互，都像在写信。理解这些「信件」的格式，你就能看懂整个系统的对话。

信的格式 — HTTP 请求解剖

想象前端要给后端写一封信，请求办一张新饭卡。这封信由四个部分组成：

1
收件地址（URL）：/api/cards

告诉后端「我要操作饭卡资源」

2
信件类型（HTTP 方法）：POST

表示「我要创建一个新东西」

3
信的内容（请求体（Body））

{ "idNumber": "202100010001", "preDeposit": 5000 }

4
回信（响应（Response））

后端返回新创建的饭卡信息，包含自动生成的卡号、余额等

信件类型决定了你想做什么。这套系统里只用四种：

GET 查询 —— 「给我看看」
POST 创建 —— 「帮我新建一个」
PUT 修改 —— 「帮我改一下」
DELETE 删除 —— 「帮我去掉」
📬
写信的隐喻

每次前端和后端的交互，就是一次「写信——回信」。信封上写着地址（URL）和类型（方法），信里装着 JSON 格式的数据。这个过程遵循 RESTful 风格。

注意金额的变化

操作员说「50 元」，但 api.js 发送的是「5000 分」。系统内部统一用「分」作为金额单位，避免小数带来的计算误差。

代码翻译 — api.js 的信使角色

api.js 就像一个专业信使。它有一个通用的「寄信」函数，所有请求都通过它发出：

api.js async function request(path, options = {}) { const url = `${BASE_URL}${path}` const res = await fetch(url, { headers: { 'Content-Type': 'application/json' }, ...options, }) const data = await res.json() if (!res.ok) { const err = new Error('请求失败') throw err } return data }
逐行翻译
async function request — 定义一个异步的「寄信」函数
url = BASE_URL + path — 拼出完整的收件地址
await fetch(url, ...) — 把信寄出去，等待回信
headers: Content-Type — 标注信的内容是 JSON 格式
res.json() — 拆开回信，读取里面的数据
if (!res.ok) — 如果回信说「出问题了」
throw err — 就大声报告错误
return data — 一切顺利，把回信内容交出来

有了通用信使，发卡请求就变得很简洁：

api.js export function issueCard({ idNumber, preDeposit = 0 }) { return request('/api/cards', { method: 'POST', body: JSON.stringify({ idNumber, preDeposit }), }) }
逐行翻译
issueCard({ idNumber, preDeposit }) — 定义「发卡」函数，接收证件号和预存金额
request('/api/cards', ...) — 让信使把信寄到 /api/cards 地址
method: 'POST' — 信封上标注「创建」类型
JSON.stringify({...}) — 把数据打包成 JSON 格式装进信封
📜
契约优先 — OpenAPI 文档

这个项目先用 OpenAPI 文档定义好所有接口规范：每个 URL 接受什么参数、返回什么数据。前后端严格按照这个契约各自开发，就像签了合同一样，双方都不能违约。这样即使两个团队同时工作也不会「对不上」。
