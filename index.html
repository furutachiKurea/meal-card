type Card struct {
    ID           uint       `gorm:"primaryKey" json:"-"`
    CardNo       string     `gorm:"uniqueIndex;not null;size:16" json:"cardNo"`
    CardHolderID uint       `gorm:"not null" json:"cardHolderId"`
    Deposit      int64      `gorm:"not null;default:0" json:"deposit"`
    Balance      int64      `gorm:"not null;default:0" json:"balance"`
    Status       CardStatus `gorm:"not null;default:'active'" json:"status"`
    CreatedAt    time.Time  `json:"createdAt"`
    UpdatedAt    time.Time  `json:"updatedAt"`
}
中文解释

type Card struct — 定义"就餐卡"表结构

ID · primaryKey · json:"-" — 内部主键，对外隐藏

CardNo · size:16 — 16位随机卡号，唯一索引

CardHolderID — 指向持卡人表的外键

Deposit int64 — 押金（分），固定 2000 = 20元

Balance int64 — 余额（分），存款加、消费减

Status — active / lost / cancelled

CreatedAt / UpdatedAt — 由 GORM 自动维护

代码翻译：消费核心逻辑

CODE 
err = s.cardRepo.DB().Transaction(func(tx *gorm.DB) error {
    txRepo := s.cardRepo.WithTx(tx)
    card, err := txRepo.FindCardByCardNo(cardNo)
    if err != nil {
        return newBizError(ErrCodeCardNotFound, "此卡非本单位所发")
    }
    if card.Status == model.CardStatusCancelled {
        return newBizError(ErrCodeCardCancelled, "此卡已注销")
    }
    if card.Status == model.CardStatusLost {
        return newBizError(ErrCodeCardLost, "此卡已挂失")
    }
    card.Balance -= amount
    txRepo.UpdateCard(card)
    txRepo.CreateTransaction(txn)
    return nil
})
中文解释

Transaction(func...) — 开启事务

WithTx(tx) — 创建绑定本次事务的仓库实例

FindCardByCardNo — 查卡，找不到 = 不是我们发的

Status == Cancelled — 已注销，拒绝

Status == Lost — 已挂失，拒绝并报警

Balance -= amount — 扣减余额（单位：分）

UpdateCard + CreateTransaction — 更新余额 + 写流水，事务保证不可分割

return nil — 成功，事务提交

两个关键设计思想

💡
事务：要么全成功，要么全回滚

扣款成功但写流水断电？有事务保护 → 扣款自动撤销 → 余额恢复。

🏗️
三层架构：各管各的

Handler 只管 HTTP · Service 只管业务规则 · Repository 只管数据库。改一层不影响其他层。

BroadcastChannel 核心代码

CODE 
// 顾客屏：刷卡后广播
const ch = new BroadcastChannel('window-1')
const res = await getCard(cardNo)
ch.postMessage({
  type: 'card_read',
  cardNo: res.cardNo,
  holderName: res.cardHolder.name,
  balance: res.balance
})

// 操作端：监听
ch.onmessage = (e) => {
  if (e.data.type === 'card_read') {
    setCardInfo(e.data)
  }
}
中文解释

new BroadcastChannel('window-1') — 打开对讲机，拨到频道 window-1

await getCard(cardNo) — 顾客屏先向后端验证卡片

ch.postMessage({...}) — 验证通过，把信息广播出去

ch.onmessage — 操作端监听，收到后更新界面

为什么要分两块屏？

⚠️
如果两边都能输入卡号？

阿姨可能手动录入 → "替人消费"漏洞。职责单一从设计上杜绝：顾客屏是唯一刷卡入口，操作端只能被动接收。

🎓

顾客屏只做

输入卡号 + 显示结果

👩‍🍳

操作端只做

输入金额 + 确认结算

系统如何保护学生的钱包

限额

单笔≤200元，日累计≤500元

💡
为什么三重而不是一重？

只查余额无法防盗刷，只查挂失无法识别注销卡。三重各司其职。

限额设计与卡号安全

💸
消费限额

单笔200元防大额盗刷 · 日累计500元防多次小额盗刷。金额以"分"存储避免浮点误差。

🔢
16位随机卡号

自增 ID（1,2,3…）可预测——知道自己是42号就能猜41、43号。16位随机数穷举概率 1/10¹⁶。
