系统全貌

这个系统是什么、解决什么问题、有哪些角色和入口

你有没有遇到过这些烦恼？

早上八点，你拿着饭卡冲进食堂，窗口阿姨刷了一下——

"余额不足，请充值。"

你懵了。昨天明明还有钱的？卡丢了怎么办？充进去的钱还能退吗？

食堂饭卡管理系统就是为了解决这一系列问题而生的。它把食堂的 六项核心业务 全部数字化，让管理员、窗口、学生都能顺畅协作。

学生刷卡 → 余额扣减 → 流水记录 一次 HTTP 请求 → 数据库事务 → 持久化日志

谁在使用这个系统？

🏢

管理员

负责发卡、为学生存款、处理挂失与注销、查看统计报表。是系统的核心操作者。

🍜

窗口阿姨

通过窗口机验卡、录入餐费、完成结算。每次就餐的最终执行者。

🎓

学生

持卡就餐，刷卡时可看到当前余额。无需登录系统，卡即身份。

想象食堂管理处有一个大型档案室，里面摆着几排档案柜。每个抽屉专门存放一类信息，抽屉之间通过编号相互引用。

🔖

这个系统一共有 5 个抽屉，每个抽屉负责一类数据，抽屉之间通过 外键 彼此关联。

读懂一张表的代码

下面是 Card（就餐卡）表在 Go 代码中的定义。左边是代码，右边是逐行的中文解释。

CODE 
type Card struct {
    ID           uint       `gorm:"primaryKey" json:"-"`
    CardNo       string     `gorm:"uniqueIndex;not null;size:16" json:"cardNo"`
    CardHolderID uint       `gorm:"not null" json:"cardHolderId"`
    Deposit      int64      `gorm:"not null;default:0" json:"deposit"`
    Balance      int64      `gorm:"not null;default:0" json:"balance"`
    Status       CardStatus `gorm:"not null;default:'active'" json:"status"`
    CreatedAt    time.Time  `json:"createdAt"`
    UpdatedAt    time.Time  `json:"updatedAt"`
}
中文解释

type Card struct — 定义"就餐卡"这张表的数据结构，每个字段对应表里的一列

ID uint · primaryKey — 数据库内部的主键，自动递增，对外接口隐藏（json:"-"）

CardNo string · size:16 — 对外暴露的 16 位随机卡号，设置唯一索引，保证不重复

CardHolderID uint — 指向持卡人表的外键，把这张卡和它的主人关联起来

Deposit int64 — 押金，单位是"分"，固定为 2000（即 20 元）

Balance int64 — 卡内余额，单位是"分"，每次存款加、消费减

Status CardStatus — 卡的状态：active（正常）/ lost（挂失）/ cancelled（注销），默认 active

CreatedAt / UpdatedAt — 创建时间和最后更新时间，由GORM 自动维护

检验一下

🧪 检验理解

1. 系统为什么用整数（int64）来存储金额，而不是小数（float）？

✅ 正确！浮点数的二进制表示会引入微小误差，在金融场景中累积后会造成账目不准。用"分"作为最小单位，全程整数运算，完全避免这个问题。
❌ 再想想。浮点数的问题是精度，不是大小。试试 B 选项——想想为什么 0.1 + 0.2 在计算机里不等于 0.3。

2. 就餐卡的 CardNo（卡号）为什么用 16 位随机数字，而不直接用数据库自增 ID？

✅ 正确！如果卡号是 0001、0002……攻击者可以轻易猜出其他人的卡号，也能推算出系统规模。随机卡号从根本上消除了这种风险。
❌ 不对。安全才是核心原因。如果卡号 = 自增 ID，别人只要知道自己的卡号，就能猜出所有其他人的卡号。

3. 一个持卡人（CardHolder）最多可以拥有几张状态为 active 的就餐卡？

✅ 正确！系统在数据层面通过唯一约束保证一人一卡，这也是需求中"持有效证件办理就餐卡"的核心规则。
❌ 不对。看看需求描述："学生员工持有效证件到管理处……办理就餐卡"——一人只能对应一张卡，系统通过证件号唯一约束来保证这一点。

代码翻译：消费核心逻辑

CreateTransaction 中最关键的一段代码。左边是 Go 代码，右边是白话解释。

CODE 
err = s.cardRepo.DB().Transaction(func(tx *gorm.DB) error {
    txRepo := s.cardRepo.WithTx(tx)
    card, err := txRepo.FindCardByCardNo(cardNo)
    if err != nil {
        return newBizError(ErrCodeCardNotFound, "此卡非本单位所发")
    }
    if card.Status == model.CardStatusCancelled {
        return newBizError(ErrCodeCardCancelled, "此卡已注销")
    }
    if card.Status == model.CardStatusLost {
        return newBizError(ErrCodeCardLost, "此卡已挂失")
    }
    card.Balance -= amount
    txRepo.UpdateCard(card)
    txRepo.CreateTransaction(txn)
    return nil
})
中文解释

Transaction(func...) — 开启数据库事务，里面的所有操作要么全部成功，要么全部撤销

WithTx(tx) — 创建一个绑定了本次事务的仓库实例，确保后续查询都在同一个事务里

FindCardByCardNo — 根据卡号查数据库，找不到说明不是我们发的卡

Status == Cancelled — 第二重校验：卡已注销，拒绝

Status == Lost — 第三重校验：卡已挂失，拒绝并报警

Balance -= amount — 扣减余额（单位：分）

UpdateCard + CreateTransaction — 同时更新余额和写入消费记录，事务保证两步不可分割

return nil — 返回 nil 表示成功，事务自动提交

两个关键设计思想

💡
事务：要么全部成功，要么全部回滚

假设扣款成功了，但写流水记录时突然断电。没有事务保护的话，余额减少了却没有记录——账就对不上了。有了事务，这两步是捆绑的：写流水失败 → 扣款自动撤销 → 余额恢复原样。

🏗️
三层架构：每一层只做自己的事

Handler 层（HTTP 入口）只负责解析请求、返回响应，不碰业务逻辑。
Service 层（业务逻辑）只负责校验规则、编排流程，不直接写 SQL。
Repository 层（数据访问）只负责与数据库打交道，不关心业务含义。
这样分层的好处是：改数据库只动 Repository，改业务规则只动 Service，互不影响。

双屏窗口机

如何用一个浏览器 API 实现两块屏幕的实时联动

两块屏幕，一条频道

想象一对对讲机——只要两台设备调到同一频率，按下说话键就能互通。不需要电话基站，不需要服务器中转，信号直接在两端之间传递。

🖥️

顾客屏

学生刷卡、查看余额、等待结算结果
/window/customer?id=1

👩‍🍳

操作端（阿姨屏）

等待刷卡、查看学生信息、输入金额结算
/window?id=1

💡
两块屏幕都在同一台电脑的浏览器里，通过 BroadcastChannel 频道 window-{id} 互相通信，完全不经过后端。

BroadcastChannel 核心代码

CODE 
// 顾客屏：刷卡后广播给操作端
const ch = new BroadcastChannel('window-1')
const res = await getCard(cardNo)
ch.postMessage({
  type: 'card_read',
  cardNo: res.cardNo,
  holderName: res.cardHolder.name,
  balance: res.balance
})

// 操作端：监听顾客屏消息
ch.onmessage = (e) => {
  if (e.data.type === 'card_read') {
    setCardInfo(e.data)
  }
}
中文解释

new BroadcastChannel('window-1') — 打开对讲机，拨到频道 window-1。两块屏幕用同一个频道名才能互通

await getCard(cardNo) — 顾客屏先向后端查卡，验证卡片状态

ch.postMessage({...}) — 验证通过后，把卡号、姓名、余额打包成一条消息，广播出去

ch.onmessage — 操作端一直"守着收音机"，收到 card_read 消息就立刻更新界面

🔑
同源限制

BroadcastChannel 只在同源页面之间生效。两块屏幕必须来自同一个前端服务，这在窗口机（同一台电脑）的场景下天然满足。

为什么要分两块屏？

⚠️
如果两边都能输入卡号会怎样？

阿姨可能在学生还没刷卡时就手动录入卡号，导致"替人消费"的安全漏洞；或者两边同时触发查卡请求，产生竞态条件。
职责单一从设计上杜绝了这类问题——顾客屏是唯一的刷卡入口，操作端只能被动接收信息。

🎓

顾客屏只做一件事

输入卡号 + 显示结果。不能输入金额，不能结算。

👩‍🍳

操作端只做一件事

输入金额 + 确认结算。不能输入卡号，必须等学生先刷。

05

安全与限额

系统如何保护学生的钱包

像银行一样的多重防线

每一笔消费，都必须通过三道"门"才能放行。就像银行取款——先刷卡、再核身份、再看余额，缺一不可。

🚪

门禁检查

卡必须存在且未被注销

🪪

柜台确认

卡未被挂失，仍可正常使用

💰

限额检查

单笔 ≤ 200元，当日累计 ≤ 500元，防止异常盗刷

💡
为什么是三重而不是一重？

单纯检查余额无法阻止被盗刷；单纯检查挂失无法识别已注销的卡。三重校验各司其职，互相补充。

限额设计与卡号安全

💸
为什么要有消费限额？

卡丢失或被复制后，单笔 200 元上限能将损失控制在可接受范围；日累计 500 元确保即使多次小额盗刷，当日总损失也有上限。

🔢
为什么卡号是 16 位随机数？

如果卡号 = 自增 ID（1, 2, 3…），攻击者知道自己是 42 号就能猜测 41、43 号。16 位随机数的穷举概率为 1/10¹⁶——在计算上不可能猜中。

检验你的理解

🧪 安全检验

1. 小明的饭卡被挂失后，他拿着卡去食堂窗口消费，会发生什么？

✅ 正确！挂失校验在余额校验之前，系统不会走到扣款步骤，直接报警。
❌ 挂失校验发生在余额校验之前，系统根本不会检查余额，而是直接触发报警。

2. 小红今天已经消费了 450 元，现在想再消费 60 元，结果会是？

✅ 正确！450 + 60 = 510 超过日限额 500，系统拒绝整笔消费。
❌ 系统不做「部分扣款」，整笔 60 元被完整拒绝。

3. 为什么用 16 位随机数字作卡号而不是数据库自增 ID？

✅ 正确！自增 ID 是可预测的，随机卡号让穷举攻击不可行。
❌ 核心原因是安全性——自增 ID 暴露了数据库结构和用户顺序。
