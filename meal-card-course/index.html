代码翻译 — 三重校验

当一张卡尝试消费时，后端并不会直接扣钱。它会先做三道检查，就像保安查验你的身份一样。下面左侧是真实代码，右侧是逐行翻译。

Go — card_service.go if card.Status == model.CardStatusCancelled { return nil, newBizError(ErrCodeCardCancelled, "此卡已注销") } if card.Status == model.CardStatusLost { return nil, newBizError(ErrCodeCardLost, "此卡已挂失") } if card.Balance < amount { return nil, newBizError(ErrCodeInsufficientBalance, "余额不足") }
逐行翻译

第 2 关：如果这张卡的状态是"已挂失" ——
  → 拒绝！返回错误信息："此卡已挂失"
 
第 3 关：如果卡里的余额 < 消费金额 ——
  → 拒绝！返回错误信息："余额不足"

什么是「防御性编程」？

防御性编程是一种编程思维：先假设一切都可能出错，在真正执行操作之前，把所有可能的异常情况都检查一遍。 就像你出门前会检查钥匙、手机、钱包一样 —— 宁可多查几遍，也不要事后才发现问题。 上面的三重校验就是典型的防御性编程：先排除所有不合法的情况，最后才执行扣款。

注意：只有三关全部通过，代码才会继续往下执行真正的扣款操作。任何一关失败，都会立刻返回错误，绝不会多扣一分钱。

为什么金额用「分」存储？

所有金额字段都用 int64（整数）存储，单位是「分」而非「元」。 100 分 = 1 元。这是软件工程中处理钱的标准做法 —— 计算机做浮点小数运算会产生误差（例如 0.1 + 0.2 ≠ 0.3），用整数就完全精确。

数据的对话 — 前后端如何通信

前端和后端之间的每一次交互，都像在写信。理解这些「信件」的格式，你就能看懂整个系统的对话。

信的格式 — HTTP 请求解剖

想象前端要给后端写一封信，请求办一张新饭卡。这封信由四个部分组成：

1
收件地址（URL）：/api/cards

告诉后端「我要操作饭卡资源」

2
信件类型（HTTP 方法）：POST

表示「我要创建一个新东西」

3
信的内容（请求体（Body））

{ "idNumber": "202100010001", "preDeposit": 5000 }

4
回信（响应（Response））

后端返回新创建的饭卡信息，包含自动生成的卡号、余额等

信件类型决定了你想做什么。这套系统里只用四种：

GET 查询 —— 「给我看看」
POST 创建 —— 「帮我新建一个」
PUT 修改 —— 「帮我改一下」
DELETE 删除 —— 「帮我去掉」
📬
写信的隐喻

每次前端和后端的交互，就是一次「写信——回信」。信封上写着地址（URL）和类型（方法），信里装着 JSON 格式的数据。这个过程遵循 RESTful 风格。

代码翻译 — api.js 的信使角色

api.js 就像一个专业信使。它有一个通用的「寄信」函数，所有请求都通过它发出：

api.js async function request(path, options = {}) { const url = `${BASE_URL}${path}` const res = await fetch(url, { headers: { 'Content-Type': 'application/json' }, ...options, }) const data = await res.json() if (!res.ok) { const err = new Error('请求失败') throw err } return data }
逐行翻译
async function request — 定义一个异步的「寄信」函数
url = BASE_URL + path — 拼出完整的收件地址
await fetch(url, ...) — 把信寄出去，等待回信
headers: Content-Type — 标注信的内容是 JSON 格式
res.json() — 拆开回信，读取里面的数据
if (!res.ok) — 如果回信说「出问题了」
throw err — 就大声报告错误
return data — 一切顺利，把回信内容交出来

有了通用信使，发卡请求就变得很简洁：

api.js export function issueCard({ idNumber, preDeposit = 0 }) { return request('/api/cards', { method: 'POST', body: JSON.stringify({ idNumber, preDeposit }), }) }
逐行翻译
issueCard({ idNumber, preDeposit }) — 定义「发卡」函数，接收证件号和预存金额
request('/api/cards', ...) — 让信使把信寄到 /api/cards 地址
method: 'POST' — 信封上标注「创建」类型
JSON.stringify({...}) — 把数据打包成 JSON 格式装进信封

卡的一生 — 状态机与业务规则

饭卡就像交通信号灯：绿灯通行，黄灯暂停，红灯永停。系统里有一个「交警」确保每张卡只能按规则变灯。

卡的三种状态

每张饭卡在任意时刻只能处于以下三种状态之一，就像交通灯不可能同时亮两种颜色。

🟢
active — 绿灯

正常使用中。可以刷卡消费、充值，一切功能正常。

🟡
lost — 黄灯

已挂失。不能消费也不能充值，但可以取消挂失恢复使用，或办新卡。

🔴
cancelled — 红灯

已注销，永久停用。押金和余额已退还，这张卡的生命到此结束。

注意 cancelled 是终态 —— 一旦注销就无法恢复。就像红灯被焊死了，再也变不回绿灯。

代码翻译 — 挂失与取消挂失

下面是 ReportLoss 方法的真实代码。左侧是 Go 代码，右侧是逐行翻译成人话。

Go func (s *CardService) ReportLoss(cardNo string) (*model.Card, error) { card, err := s.cardRepo.FindCardByCardNo(cardNo) if err != nil { if errors.Is(err, gorm.ErrRecordNotFound) { return nil, newBizError("CARD_NOT_FOUND", "卡号不存在") } return nil, err } if card.Status != model.CardStatusActive { return nil, newBizError("CARD_NOT_ACTIVE", "只有正常使用中的卡可以挂失") } card.Status = model.CardStatusLost if err := s.cardRepo.UpdateCard(card); err != nil { return nil, err } return card, nil }
逐行翻译
定义挂失方法：接收一个卡号，返回修改后的卡片或错误信息。
第一步：找卡。用卡号去数据库里查这张卡。
如果查询出错了……
特别地，如果根本找不到这个卡号——
返回一个业务错误：「卡号不存在」。
（其他数据库错误，直接返回。）
 
 
第二步：检查状态。如果卡不是 active（绿灯）——
拒绝操作，返回：「只有绿灯状态的卡才能挂失」。
 
第三步：变灯。把状态从 active（绿灯）改成 lost（黄灯）。
保存到数据库。如果保存失败，返回错误。
 
 
成功！返回更新后的卡片信息。
 
🚦
模式：先检查再操作

每个状态变更方法都遵循相同的三步：找到卡 → 检查当前状态是否允许 → 执行变更。这就是「交警」执法的标准流程。

发卡的特殊规则 — 一人一卡

发卡看似简单，其实藏着一条复杂的业务规则链。系统必须处理各种「已有旧卡」的情况。

1
验证证件号

调用学籍验证服务，确认这个证件号对应的是本单位的学生或员工。

2
检查 active 卡 → 拒绝

如果该证件号已有一张绿灯（active）卡，直接拒绝：「该证件号已有正在使用的卡」。一人一卡，不能同时持有两张有效卡。

3
检查 lost 卡 → 自动注销旧卡

如果有一张黄灯（lost）卡，系统会自动注销它，退还旧卡的押金和余额，然后继续发新卡。

4
生成 16 位随机卡号

用随机数生成器创建一个 16 位的唯一卡号，类似银行卡号。

5
创建新卡

押金固定 2000 分（即 20 元），预存款记入卡中余额，状态设为 active（绿灯亮起）。

🧠
概念：状态机

饭卡的 active / lost / cancelled 就构成了一个状态机。代码里用 枚举 定义了三种合法状态，每个方法在操作前都会检查当前状态是否允许此操作。这是软件中管理复杂业务规则的核心模式 —— 让代码成为规则的执行者，而不是靠人来记住规则。

注销的退款逻辑

注销是饭卡的「最终告别」。关键在于：先拍照留存退款金额，再清零余额、改状态。

Go func (s *CardService) CancelCard(cardNo string) (*CancellationResult, error) { card, err := s.cardRepo.FindCardByCardNo(cardNo) if err != nil { if errors.Is(err, gorm.ErrRecordNotFound) { return nil, newBizError("CARD_NOT_FOUND", "卡号不存在") } return nil, err } if card.Status == model.CardStatusCancelled { return nil, newBizError("CARD_ALREADY_CANCELLED", "该卡已注销") } deposit := card.Deposit balance := card.Balance card.Status = model.CardStatusCancelled card.Balance = 0 if err := s.cardRepo.UpdateCard(card); err != nil { return nil, err } return &CancellationResult{ Card: card, Deposit: deposit, Balance: balance, Total: deposit + balance, }, nil }
逐行翻译
定义注销方法：接收卡号，返回退款详情或错误。
第一步：找卡。老套路，先查出这张卡。
如果查询出错……
找不到这个卡号？
返回错误码 CARD_NOT_FOUND。
 
 
 
第二步：检查状态。如果已经是红灯（cancelled）——
拒绝：「已经注销过了，不能重复注销」。
 
第三步：拍照！先把押金存到 deposit 变量里。
再把余额存到 balance 变量里。这就是快照。
第四步：变灯。把状态改为 cancelled（红灯永亮）。
余额清零 —— 钱已经准备退还了。
保存到数据库。
 
 
第五步：返回退款清单。
卡片信息 + 押金快照 + 余额快照。
总退款 = 押金 + 余额。打包返回。
 
 
📸
为什么要「先拍照再清零」？

如果先清零再去读金额，读到的就是 0 了 —— 系统就不知道该退多少钱。这种「先保存旧值，再修改」的技巧在编程中非常常见，就像搬家前先拍照留证。

分层架构的好处

每一层只和相邻的层打交道：Handler 不直接碰数据库，Service 不知道 HTTP 长什么样。这样改动某一层时，其他层不受影响。就像水务系统中，换了水龙头不需要改水厂。

Screen 2 · 统计的秘密 — 聚合查询

统计服务不创建新数据，它只是从已有的消费记录和充值记录中聚合出数字。下面是「日餐报表」功能的核心代码。

Go func (s *StatisticsService) GetDailyReport(date string) (*DailyReportResult, error) { day, err := time.Parse("2006-01-02", date) if err != nil { return nil, newBizError("日期格式无效") } start := time.Date(day.Year(), day.Month(), day.Day(), 0,0,0,0, time.Local) end := start.Add(24 * time.Hour) totalRevenue, count, windows, err := s.cardRepo.GetDailyReport(start, end) }
逐行解读
第 1 行：定义函数 —— 接收一个日期字符串，返回报表结果
第 2-4 行：把日期文字转换成时间对象；如果格式不对就报错
第 5-6 行：计算这一天的起点（0:00）和终点（次日 0:00）
第 7 行：把时间范围交给 Repository，让它去数据库查出总收入、笔数和各窗口数据

什么是聚合查询？

统计不是把每条数据一条条读出来手动加总，而是让数据库用 SUM、 COUNT、 GROUP BY 一次性算好。就像问全班同学的平均分 —— 你不需要挨个问成绩再手动计算，直接让数据库帮你算。

Screen 3 · 代码翻译 — 前端怎么发 API 请求

前端通过 api.js 中的函数向后端发起请求。每个统计功能对应一个函数。

JavaScript export function getMealRevenue({ startTime, endTime }) { return request( `/api/statistics/meal-revenue ?startTime=${encodeURIComponent(startTime)} &endTime=${encodeURIComponent(endTime)}` ) } export function getActiveBalance() { return request('/api/statistics/active-balance') }
逐行解读
getMealRevenue：获取指定时间段的售饭总收入。通过查询参数告诉后端时间范围
?startTime=...&endTime=...：就像信封上的附加说明 —— 「我只要这个时间范围内的数据」
encodeURIComponent：把时间中的特殊字符（如冒号 :）转成安全格式，避免 URL 被破坏
getActiveBalance：获取流动资金总额。不需要参数 —— 直接查所有活跃卡的余额之和

为什么统计接口都用 GET？

HTTP 中 GET 代表「获取数据」，POST 代表「创建/修改数据」。统计功能只是读取已有数据做汇总，不修改任何东西，所以全部使用 GET 请求。就像去图书馆查资料（GET）和往图书馆捐书（POST）是不同的动作。

系统统计模块的全部 API 一览：

GET /meal-revenue 指定时段售饭总收入
GET /window-revenue 各窗口收入分布
GET /holder-deposits 持卡人存款明细（分页）
GET /active-balance 流动资金总额
GET /daily-report 日餐报表
GET /yearly-report 年餐报表（按月汇总）
