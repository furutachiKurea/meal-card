认识演员们 — 系统中的「角色」

每部电影都有一群各司其职的演员。软件系统也一样 —— 让我们认识这个饭卡系统里的「角色」们。

Screen 1 / 4 — 前端：用户看到的界面

把系统想象成一家医院。前端就是前台大厅 —— 患者（用户）走进来看到的一切：挂号窗口、指示牌、叫号屏幕。 前端项目的文件结构长这样：

frontend/src/
App.jsx — 路由总控，决定显示哪个页面
api.js — 「信使」，负责与后端通信
layouts/
AdminLayout.jsx — 管理端的外壳（侧边栏 + 内容区）
pages/ — 每个 JSX 文件 = 一个页面
IssuePage.jsx — 发卡页面
DepositPage.jsx — 存款页面
MealPage.jsx — 就餐消费（窗口机）
LossPage.jsx — 挂失管理
CancelPage.jsx — 注销退卡
StatisticsPage.jsx — 汇总统计
WindowsPage.jsx — 窗口管理

前端分为两个「世界」：

🏥
管理端

AdminLayout 包裹 6 个功能页面（发卡、存款、挂失、注销、统计、窗口管理），带侧边栏导航。就像医院的行政办公区。

🍽️
窗口机端

MealPage 独立运行，全屏深色界面，模拟食堂刷卡组件。就像医院的自助挂号机。

Screen 2 / 4 — 后端：三层架构

后端就是医院的「幕后团队」。它被组织成三层，各管各的事情，互不干扰。

👩‍⚕️
Handler（护士站）

接收 HTTP 请求，解析参数，返回 JSON 响应。 位置：backend/handler/

👨‍⚕️
Service（医生）

核心业务逻辑：校验规则、状态流转、余额计算。系统的「大脑」。 位置：backend/service/

📁
Repository（病历档案室）

数据库读写操作，SQL 查询封装。所有数据的「保险柜」。 位置：backend/repository/

三层是怎么「组装」起来的？看 main.go 里的依赖注入代码：

Go // 1. 创建 Repository（档案室） cardRepo := repository.NewCardRepository(gormDB) windowRepo := repository.NewWindowRepository(gormDB)   // 2. 创建验证器 studentValidator := client.NewMockStudentValidator()   // 3. 创建 Service（医生），把档案室交给它 cardSvc := service.NewCardService(cardRepo, windowRepo, studentValidator)   // 4. 创建 Handler（护士站），把医生交给它 cardH := handler.NewCardHandler(cardSvc, studentValidator, cardRepo)
翻译成人话
先准备好病历档案室（Repository），告诉它去哪个数据库拿数据。
再准备一个身份验证器，用来核查学生证件是否有效。
然后雇一位医生（Service），把档案室的钥匙和验证器交给他。
最后安排一位护士（Handler），告诉她有问题去找哪位医生。
💡
软件设计原则：关注点分离（Separation of Concerns）

每一层只管自己的事。护士不开药方，医生不管档案归档，档案员不给病人问诊。改动一层时，其他层不受影响。

Screen 3 / 4 — 数据模型：5 张表

数据库就是医院的病历档案室。所有信息分门别类存放在 5 张表中。 我们使用 SQLite 数据库，通过 ORM 工具 GORM 来操作。

👤
CardHolder

持卡人 — 存储身份信息：姓名、证件号。每个人是一条记录，有唯一 主键 ID。

💳
Card

饭卡 — 16 位卡号、余额、押金、状态（正常/挂失/注销）。通过 外键关联到持卡人。

💰
DepositRecord

存款记录 — 每次充值一条记录，记录金额、时间。

📝
Transaction

消费记录 — 每次刷卡吃饭一条记录，记录消费金额和窗口信息。

🖥️
Window

窗口 — 食堂的每个售饭窗口，记录窗口名称和状态。

💲
为什么金额用「分」存储？

所有金额字段都用 int64（整数）存储，单位是「分」而非「元」。 100 分 = 1 元。这是软件工程中处理钱的标准做法 —— 计算机做浮点小数运算会产生误差（例如 0.1 + 0.2 ≠ 0.3），用整数就完全精确。
