刷卡就餐 — 从你熟悉的操作开始

把这个系统想象成一台自动售货机 —— 你插卡，机器验证，扣钱，出货。每一步背后都有代码在默默工作。

这个系统是什么？

食堂饭卡管理系统是学校食堂的电子钱包。学生和员工拿着一张饭卡，就能完成充值、消费、挂失等所有操作。 整个系统由 前端（操作界面）和 后端（核心逻辑）配合完成。

它一共有 6 大功能模块：

💳
发卡

持有效证件办理饭卡，缴押金，预存款项，系统自动生成唯一卡号。

💰
存款

余额不足时到管理处充值，可打印收据。

🍜
就餐消费

刷卡→验证→显示余额→输入金额→扣款，整个流程几秒完成。

📊
汇总统计

统计每日、每月的售饭收入，各窗口流水，持卡人存款明细等报表。

🚨
挂失

饭卡丢失后立即挂失，所有窗口机会拒绝已挂失的卡，保护资金安全。

❌
注销

离校或不再就餐时注销饭卡，退还押金，卡号可回收再用。

代码翻译 — 三重校验

当一张卡尝试消费时，后端并不会直接扣钱。它会先做三道检查，就像保安查验你的身份一样。下面左侧是真实代码，右侧是逐行翻译。

Go — card_service.go if card.Status == model.CardStatusCancelled { return nil, newBizError(ErrCodeCardCancelled, "此卡已注销") } if card.Status == model.CardStatusLost { return nil, newBizError(ErrCodeCardLost, "此卡已挂失") } if card.Balance < amount { return nil, newBizError(ErrCodeInsufficientBalance, "余额不足") }
逐行翻译
第 1 关：如果这张卡的状态是"已注销" ——
  → 拒绝！返回错误信息："此卡已注销"
 
第 2 关：如果这张卡的状态是"已挂失" ——
  → 拒绝！返回错误信息："此卡已挂失"
 
第 3 关：如果卡里的余额 < 消费金额 ——
  → 拒绝！返回错误信息："余额不足"

什么是「防御性编程」？

防御性编程是一种编程思维：先假设一切都可能出错，在真正执行操作之前，把所有可能的异常情况都检查一遍。 就像你出门前会检查钥匙、手机、钱包一样 —— 宁可多查几遍，也不要事后才发现问题。 上面的三重校验就是典型的防御性编程：先排除所有不合法的情况，最后才执行扣款。

注意：只有三关全部通过，代码才会继续往下执行真正的扣款操作。任何一关失败，都会立刻返回错误，绝不会多扣一分钱。
